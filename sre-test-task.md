# Тестовое задание по SRE: Kyverno MVP

Источник: PDF с исходной формулировкой тестового задания.

Компания: MTS True Tech

## Цель

- [ ] Развернуть пилотный стенд Kyverno MVP для защиты Kubernetes-контура от некорректных и опасных конфигураций.
- [ ] Наглядно продемонстрировать работу Kyverno в условиях, близких к реальным, перед полноценным внедрением в production.

## Правила выполнения и сдачи

- [ ] Выполнить тестовое задание самостоятельно.
- [ ] Выложить решение в GitHub или другую систему хранения кода.
- [ ] Передать ссылку на решение команде, которая проводит тестирование.
- [ ] Быть готовым объяснить решение на собеседовании.

## Окружение проверяющего

- [ ] Учесть, что у проверяющего есть локальный ПК, то есть основной рабочий хост.
- [ ] Учесть, что у проверяющего есть чистая виртуальная машина Debian или Ubuntu в той же сети.
- [ ] Ориентироваться на VM с 8 ГБ RAM, но решение должно запускаться и работать при допустимых отличиях по памяти.
- [ ] Сделать так, чтобы инфраструктурный код успешно разворачивался с локальной машины проверяющего на виртуальную машину.
- [ ] В результате развертывания должен получаться готовый рабочий стенд.

## Что проверяющий может сделать вручную

- [ ] Прописать IP-адреса и домены в `/etc/hosts`, если сервисам нужны красивые URL.
- [ ] Добавить SSL/TLS-сертификаты в систему, чтобы браузер или консоль не ругались на незащищенное соединение.
- [ ] Скопировать или форкнуть код в свой аккаунт, если инфраструктура должна автоматически подтягивать изменения из Git.
- [ ] Создать собственные пароли и API-токены и передать их скрипту, плейбуку или другому механизму запуска.
- [ ] Выполнить вручную действия, явно описанные в `README.md`, чтобы убедиться в работоспособности Kyverno.

## Что проверяющий не должен додумывать

- [ ] Не требовать от проверяющего самостоятельно решать проблемы с доступами или конфигурацией.
- [ ] Описать все необходимые действия в `README.md`.

## Обязательные критерии приемки

- [ ] Kubernetes развернут.
- [ ] В Kubernetes развернут GitOps-компонент, например ArgoCD, FluxCD или аналог.
- [ ] В Kubernetes развернут контроллер для управления входящим трафиком.
- [ ] В Kubernetes развернуты Prometheus и Grafana.
- [ ] В Kubernetes развернут Kyverno.
- [ ] Существует репозиторий с несколькими политиками Kyverno.
- [ ] Для всех политик в репозитории настроены автоматические тесты, например через `kyverno test`.
- [ ] Настроен процесс, блокирующий раскатку политик, если они не прошли тесты.
- [ ] Видна реальная работа политик: кластер отклоняет деплой манифестов, нарушающих правила безопасности.
- [ ] Настроена визуализация состояния безопасности кластера:
  - [ ] метрики Kyverno;
  - [ ] статистика срабатывания правил;
  - [ ] специализированные дашборды;
  - [ ] Policy Reporter или аналогичный инструмент.

## Бонус

- [ ] Любые улучшения и расширения функционала приветствуются.

## Рекомендации по технологическому стеку

### Kubernetes

- [ ] Рассмотреть запуск Kubernetes-кластера на единственной виртуальной машине.
- [ ] Можно использовать другой способ запуска Kubernetes, если стенд работает.

### GitOps

- [ ] Использовать GitOps-подход для сокращения количества ручных операций при поднятии стенда.
- [ ] Если выбран ArgoCD, рассмотреть подход App of Apps.
- [ ] Подготовить мета-приложение, которое автоматически подтягивает остальные приложения в ArgoCD из репозиториев.

### Автоматизация

- [ ] Рассмотреть Ansible для подготовки VM и инициализации GitOps-движка.
- [ ] Стремиться к тому, чтобы стенд поднимался одной командой с локального ПК.

### Входящий трафик

- [ ] Предусмотреть точку входа, чтобы проверяющий мог открыть дашборды по красивым URL в браузере.
- [ ] Учитывать, что классический ingress-nginx постепенно уступает место более современным и безопасным решениям.
- [ ] Рассмотреть Envoy или envoy-подобные решения.
- [ ] Учитывать, что современные Helm-чарты инфраструктурных приложений обычно адаптированы под envoy-подобные решения.

### Observability

- [ ] Рассмотреть kube-prometheus-stack для развертывания observability-стека.
- [ ] Если используется kube-prometheus-stack, отключить лишнее для минимизации использования ресурсов стенда.

### CI

- [ ] Рассмотреть GitHub Actions для организации тестирования политик.
